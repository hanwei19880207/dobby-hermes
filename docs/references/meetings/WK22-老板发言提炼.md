## 二、老板的核心要求

### 1. 所有人必须动起来

> "我希望全部动起来，至少大的域交易、财经、端、算法、数据库，我希望所有域都动起来。你至少让我每次都会有人上去分享。你看一下谁不动，那就说明谁在这方面还是不够。"

**含义**：每个业务域（包括质量域）都要有 AI Native 实践产出，每周轮着上台分享。不动就是落后。

### 2. 架构必须画清楚

> "你下次这个东西把系统架构划划清楚。我最烦内部搞不清楚哪些是系统、哪些是所谓的画个模块出来的。"

**含义**：汇报时必须讲清楚系统边界。模型网关、MaaS平台、model管理界面之间的关系必须清晰。不许用模糊的"模块"来糊弄。

### 3. AI Native 要解决协同低效

> "这个方式的好处就是可以解决我们之前整个协同的问题，协同非常低下的问题，又可以解决大家每个域可以自己主动去试。"

**含义**：Oxygen SE（Zero + One）的核心价值之一是解决跨域协同低效。这与你的主测试痛点直接相关——联调协作低效正是跨域协同的典型场景。

### 4. 先野蛮生长再收口

> "初期就是上次喜哥说的问题，我们是鼓励大家先跑起来，还是说逐步收口？那可能初期反正大家先跑。"
> "我非常鼓励。但我要鼓励的话，一个点的话是完全 AI 原生的，就是完全是走向 AI 原生。但你们这个只是页面型的东西，我要的是能够快。"

**含义**：短期鼓励各域自由探索（野蛮生长），长期会做标准收口（2~3个标准Agent）。但探索必须是真AI Native，不能只是页面开发提效——要有更深层的价值。

### 5. 最终要收口到 2~3 个标准

> "到一这个时候，我们肯定会去做一些相应的收口。你不可能说每个部门都不一样，对不对？那也很难，我们的插件就很难往下做了。"
> "我有 2~3 个标准的就差不多了。"

**含义**：各域可以先用不同的 Agent（Codex / JoyCode / Cursor），但最终要收敛到 2~3 个标准。这是给 Harness 工程的明确信号——工程实践最终要有统一标准。

### 6. 每个域要有自己的特殊性

> "充分的能够认同每个域，它自己有自己的特殊性。但是至少解决了以前相应的所谓协作当中的全局性的问题。"

**含义**：全局知识/Skill + 各域个性化拼在一起。这正是你主测试 Agent 的定位——全局协调 + 各域个性化。

### 7. SQL 生成不是难事，难在上下文

> "SQL 这个肯定是有机会的。但 SQL 的最大的问题点的话是它上下文不行，就我说跟工程的上下文很难集合。"

**含义**：纯代码生成不是壁垒，真正的难点是把 SQL 和工程上下文（表结构变更、业务语义）结合起来。这跟质量域用例生成面临的问题一样——难的不是生成文本，而是理解业务上下文。

### 8. 数据团队应该主动串联

> "数据肯定是数据团队最熟悉。你最后从代码往前做的话，现在好的话，就是前台前面把数据模型 schema、变更带来的表结构变更都给你传过去了。你要做的话是这个表结构变更对你的数据架构有什么影响。以前可能说这个东西你是很难去获取的，因为有获取的话，还是通过流程，通过人去 check，现在就全自动化了。"

**含义**：上游变更信息现在已经自动化传递了，各域应该主动串联。质量域同理——联调变更信息应该自动流转到测试侧。

## 三、老板的核心观点

### 关于 AI Native 组织推进

| 观点 | 原文摘要 |
|------|---------|
| 全域动员 | "每个域都会有人上去分享。你看一下谁不动" |
| 野蛮生长→收口 | "鼓励大家先跑起来" → "2~3个标准" |
| 共享+个性 | "全局生成之后，你再加上你自己个性化的拼起来" |
| 解决协同痛点 | "解决以前协作当中的全局性问题" |

### 关于技术方向

| 观点 | 原文摘要 |
|------|---------|
| 架构清晰是底线 | "我最烦内部搞不清楚哪些是系统" |
| Agent可替换 | "把 CI/CD 所有环境都打通，到时候你可以换任何的 Agent" |
| AI原生 > 页面提效 | "我要的是完全 AI 原生的" |
| 上下文是核心壁垒 | "SQL 跟工程上下文很难集合" |

### 关于团队分享机制

| 观点 | 原文摘要 |
|------|---------|
| 每周轮转 | "后面每周都会有团队来分享一下" |
| 先跑起来再说 | "毛钱的一个分享"（闽南语/口语：初步的分享） |
| 不动就落后 | "谁不动，那就说明谁在这方面还是不够" |
